# AI Agent 基础概念

## 什么是 AI Agent？

AI Agent（人工智能代理）是一个**自主的、目标驱动的系统**，由大语言模型（LLM）驱动，能够通过工具与环境交互，代表用户完成任务。

### Agent vs 传统聊天机器人

| 特性 | 传统聊天机器人 | AI Agent |
|------|---------------|----------|
| 交互模式 | 单轮问答 | 多轮自主执行 |
| 能力边界 | 仅生成文本 | 可调用工具、执行动作 |
| 记忆 | 无或有限 | 短期 + 长期记忆 |
| 目标导向 | 被动响应 | 主动规划与执行 |
| 环境交互 | 无 | 可读取文件、调用 API、执行命令 |

### 定义之争

关于什么是"Agent"，业界没有统一的标准定义：

- **OpenAI**：Agent 是"能够智能地完成任务的系统，从执行简单工作流到追求复杂、开放式目标"
- **Anthropic**：区分了 Agent 和 Workflow——Workflow 是"通过预定义代码路径编排 LLM 和工具的系统"，而 Agent 具有更多的自主决策能力
- **社区观点**："Agent 就是一个 LLM 在循环中搞破坏"——虽然幽默，但抓住了本质：**LLM + 工具 + 循环**

## Agent 的核心组件

一个完整的 AI Agent 系统通常包含以下核心组件：

```
┌─────────────────────────────────────────────────────────────┐
│                        AI Agent                              │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │   LLM 核心   │  │   记忆系统   │  │     工具集          │  │
│  │  (大脑)     │  │  (上下文)    │  │   (手脚)           │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│         │                │                    │              │
│         └────────────────┼────────────────────┘              │
│                          ▼                                   │
│              ┌─────────────────────┐                        │
│              │     编排控制器      │                        │
│              │   (Agent Loop)      │                        │
│              └─────────────────────┘                        │
└─────────────────────────────────────────────────────────────┘
```

### 1. LLM 核心（大脑）

LLM 是 Agent 的"大脑"，负责：
- **理解用户意图**：解析用户输入，理解任务目标
- **推理与规划**：分解复杂任务，制定执行计划
- **决策**：决定下一步行动（直接回答 vs 调用工具）
- **生成输出**：产生最终响应或工具调用指令

**模型选择考量：**
- 推理能力：复杂任务需要更强的推理模型
- 工具调用：需要支持 Function Calling
- 上下文长度：长对话需要更大的上下文窗口
- 延迟与成本：平衡性能和费用

### 2. 记忆系统（上下文）

记忆系统让 Agent 能够保持连贯的对话和任务状态：

**短期记忆（工作记忆）：**
- 当前对话历史
- 任务执行状态
- 临时变量和中间结果

**长期记忆（持久化）：**
- 用户偏好和历史交互
- 知识库和文档
- 向量数据库（语义搜索）

```python
# 记忆系统示例
class Memory:
    def __init__(self):
        self.short_term = []  # 对话历史
        self.long_term = VectorStore()  # 向量数据库
    
    def add_message(self, role, content):
        self.short_term.append({"role": role, "content": content})
    
    def search_relevant(self, query, k=5):
        """搜索相关历史信息"""
        return self.long_term.similarity_search(query, k=k)
```

### 3. 工具集（手脚）

工具是 Agent 与外部世界交互的接口：

**常见工具类型：**

| 类型 | 示例 | 用途 |
|------|------|------|
| 文件操作 | read_file, write_file | 读写文件系统 |
| 代码执行 | run_python, run_bash | 执行代码和命令 |
| 网络访问 | web_search, http_request | 获取网络信息 |
| 数据库 | query_sql, insert_data | 数据库操作 |
| API 调用 | send_email, create_issue | 外部服务集成 |

**工具定义格式：**

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_web",
            "description": "搜索互联网获取实时信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": "搜索关键词"
                    }
                },
                "required": ["query"]
            }
        }
    }
]
```

### 4. 编排控制器（Agent Loop）

编排控制器是 Agent 的"心脏"，管理整个执行循环：

```python
def agent_loop(user_message, tools, max_steps=10):
    messages = [{"role": "user", "content": user_message}]
    
    for step in range(max_steps):
        # 1. 调用 LLM
        response = llm.call(messages, tools=tools)
        
        # 2. 检查是否需要调用工具
        if response.tool_calls:
            for tool_call in response.tool_calls:
                # 3. 执行工具
                result = execute_tool(tool_call.name, tool_call.args)
                # 4. 将结果加入上下文
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": str(result)
                })
        else:
            # 5. 无工具调用，返回最终答案
            return response.content
    
    return "达到最大步数限制"
```

## Agent 的五种类型

根据能力和复杂度，Agent 可以分为五种类型：

### 1. 反应式 Agent（Reactive Agent）

最简单的 Agent 类型，单轮响应，几乎没有记忆。

```
用户输入 → LLM 处理 → 直接输出
```

**适用场景：** 简单问答、分类、翻译

### 2. 对话式助手（Conversational Assistant）

多轮对话，具有短期记忆，能记住当前会话内容。

```
用户输入 + 对话历史 → LLM 处理 → 输出
```

**适用场景：** 客服机器人、个人助手

### 3. 工具使用 Agent（Tool-Using Agent）

能够自主决定何时调用工具，获取外部信息或执行操作。

```
用户输入 → LLM 决策 → 调用工具 → 获取结果 → 生成回答
```

**适用场景：** 研究助手、数据查询、任务自动化

### 4. 规划型 Agent（Planning Agent）

能够分解复杂任务，制定计划，逐步执行并调整。

```
用户输入 → 任务分解 → 制定计划 → 执行步骤 → 验证结果 → 调整计划
```

**适用场景：** 复杂研究、项目管理、代码开发

### 5. 多 Agent 系统（Multi-Agent System）

多个专门的 Agent 协作完成复杂任务。

```
用户输入 → 编排器 → Agent A → Agent B → Agent C → 汇总结果
```

**适用场景：** 复杂业务流程、多领域协作

## Agent 工作原理：核心循环

Agent 的核心是一个"感知-思考-行动"的循环：

```
┌─────────────────────────────────────────────────────────────┐

│         ▼                                                   │
│    ┌──────────┐     ┌──────────┐                           │
│    │  感知     │ ──▶ │  思考     │                          │
│    │(理解意图) │     │(推理规划) │                          │
│    └──────────┘     └────┬─────┘                           │
│                          │                                  │
│              ┌───────────┴───────────┐                      │

│             │               ┌──────▼──────┐                 │
```

### 执行流程详解

**1. 感知阶段（Perception）**
- 接收用户输入
- 加载相关上下文（对话历史、记忆搜索结果）
- 解析用户意图

**2. 思考阶段（Reasoning）**
- LLM 进行推理
- 决定下一步行动：
  - 直接生成回答
  - 调用一个或多个工具
  - 请求更多信息

**3. 行动阶段（Action）**
- 如果决定调用工具：执行工具，获取结果
- 如果决定回答：生成最终输出

**4. 观察阶段（Observation）**
- 将工具执行结果加入上下文
- 评估是否完成任务
- 如果未完成，返回思考阶段

## 关键概念：上下文工程

构建 Agent 时，最关键的技能是**上下文工程（Context Engineering）**——构建动态系统，以正确的格式提供正确的信息和工具，使 LLM 能够合理地完成任务。

### 上下文工程的挑战

**1. 系统提示词设计**
- 太短：模型缺乏指导，行为不可预测
- 太长：模型可能困惑或忽略关键指令
- 需要平衡：清晰、简洁、全面

**2. 状态追踪**
- 当前工作目录
- 已发现的文件
- 已执行的操作
- 错误和重试状态

**3. 上下文管理**
- 保持足够的对话历史
- 避免超出 token 限制
- 处理上下文衰减（Context Rot）

**4. 工具设计**
- 工具数量：太多导致决策困难，太少能力不足
- 工具描述：清晰说明功能和参数
- 错误处理：优雅地处理工具执行失败

### 上下文衰减（Context Rot）

研究表明，当上下文过长时，模型对早期信息的关注度会下降：

```
上下文长度 vs 信息利用率

利用率 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░  短上下文（高利用率）
        ▓▓▓▓▓▓▓░░░░░░░░░░░░░░  长上下文（早期信息衰减）
        ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  压缩后（重新聚焦）
```

**解决方案：**
- 定期压缩对话历史
- 使用向量搜索检索相关信息
- 分层记忆结构

## Agent 与 Workflow 的区别

Anthropic 区分了 Agent 和 Workflow：

| 特性 | Workflow | Agent |
|------|----------|-------|
| 控制流 | 预定义的代码路径 | LLM 自主决策 |
| 灵活性 | 固定的执行顺序 | 动态调整执行路径 |
| 可预测性 | 高，易于调试 | 低，行为可能变化 |
| 适用场景 | 结构化任务 | 开放式任务 |
| 实现复杂度 | 较低 | 较高 |

**何时使用 Agent？**
- 任务路径不确定
- 需要根据中间结果调整策略
- 涉及多个工具的组合使用
- 需要处理意外情况

**何时使用 Workflow？**
- 任务步骤明确
- 需要可预测的执行
- 追求稳定性和可靠性
- 审计和合规要求高

## 小结

AI Agent 的本质是：**LLM + 工具 + 循环**

- **LLM** 提供推理和决策能力
- **工具** 提供与外部世界交互的能力
- **循环** 让 Agent 能够迭代执行直到完成任务

核心挑战不在于 Agent 循环本身（这部分很简单），而在于：
1. **上下文工程**：如何在正确的时间提供正确的信息
2. **工具设计**：如何设计有用且安全的工具
3. **安全控制**：如何限制 Agent 的行为边界
4. **评估与监控**：如何衡量和改进 Agent 性能
