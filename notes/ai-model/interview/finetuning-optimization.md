# 微调与优化面试题

本章节整理了大模型微调与优化相关的面试题目，涵盖 SFT、RLHF、LoRA、量化、推理加速等核心技术。

## 一、监督微调（SFT）

### Q1: 什么是 SFT？为什么需要 SFT？

**基础回答：**

SFT（Supervised Fine-Tuning）是在预训练模型基础上，使用标注数据进行有监督训练，使模型适应特定任务或领域。

**深入回答：**

**预训练 vs SFT**：

```
预训练:
- 目标: 学习通用语言知识
- 数据: 海量无标注文本
- 方法: 自回归语言建模

SFT:
- 目标: 学习任务特定能力
- 数据: 高质量指令数据
- 方法: 条件生成（输入→输出）
```

**追问：SFT 解决了什么问题？**

1. **对齐人类意图**：预训练模型只学会"续写"，SFT 学会"遵循指令"
2. **格式统一**：让模型输出符合预期格式
3. **能力注入**：注入特定领域能力
4. **安全性**：减少有害输出

**追问：SFT 数据的格式是什么？**

```json
{
  "instruction": "将下面的句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today"
}
```

训练时将 instruction 和 input 拼接作为模型输入，output 作为目标输出。

---

### Q2: SFT 数据如何构建？

**基础回答：**

SFT 数据可以通过人工标注、模型生成、现有数据集转换等方式构建。

**深入回答：**

**数据来源**：

| 来源 | 优点 | 缺点 |
|------|------|------|
| **人工标注** | 质量高、可控 | 成本高、速度慢 |
| **模型生成** | 规模大、成本低 | 质量参差、可能有偏见 |
| **数据集转换** | 已有标注、量大 | 格式需要适配 |
| **用户反馈** | 真实场景 | 噪声多、需清洗 |

**追问：Self-Instruct 方法是什么？**

```
Self-Instruct 流程:
1. 给定少量人工种子任务
2. 用大模型生成新任务和指令
3. 用大模型生成对应回答
4. 过滤低质量数据
5. 迭代扩充数据集
```

**追问：如何评估 SFT 数据质量？**

1. **人工抽检**：随机抽样人工审核
2. **模型评估**：用强模型评估回答质量
3. **多样性**：任务类型、指令长度的多样性
4. **难度分布**：简单/中等/困难的比例

---

### Q3: SFT 训练有哪些注意事项？

**基础回答：**

SFT 训练需要注意学习率、训练轮数、数据质量、过拟合等问题。

**深入回答：**

**关键超参数**：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **学习率** | 1e-5 ~ 5e-5 | 比预训练小 1-2 个数量级 |
| **训练轮数** | 1-3 轮 | 过多容易过拟合 |
| **Batch Size** | 32-128 | 根据显存调整 |
| **Warmup** | 3-5% | 稳定训练初期 |

**追问：为什么 SFT 学习率要比预训练小？**

1. **预训练模型已经学到了通用知识**：不需要大幅修改
2. **数据规模小**：大学习率容易过拟合
3. **灾难性遗忘**：大学习率可能破坏预训练知识

**追问：如何防止 SFT 过拟合？**

```python
# 1. 早停
if val_loss > best_loss + threshold:
    stop_training()

# 2. 学习率衰减
scheduler = get_linear_schedule_with_warmup()

# 3. 数据增强
augmented_data = augment_instructions(data)

# 4. 正则化
loss = ce_loss + weight_decay * ||θ||²
```

## 二、参数高效微调（PEFT）

### Q4: 什么是 LoRA？原理是什么？

**基础回答：**

LoRA（Low-Rank Adaptation）通过在预训练权重旁添加低秩矩阵来实现高效微调，冻结原始参数，只训练少量新增参数。

**深入回答：**

**核心公式**：

```
原始: W ∈ R^(d×k)
LoRA: W' = W + ΔW = W + BA

其中:
- B ∈ R^(d×r), A ∈ R^(r×k)
- r << min(d, k), 典型值 r=8, 16, 32
- 初始化: A ~ N(0, σ²), B = 0
```

**参数量对比**：

```
全参数微调: d × k 参数
LoRA: 2 × d × r 参数 (r << d, k)

例如 d=4096, k=4096, r=16:
- 全参数: 16M
- LoRA: 128K (减少 99%)
```

**追问：为什么初始化 B=0, A 随机？**

```
初始化时:
ΔW = BA = 0 × random = 0

这样:
- 模型初始行为与预训练完全一致
- 训练逐渐学习任务特定调整
- 避免破坏预训练知识
```

**追问：LoRA 应该加在哪些层？**

| 位置 | 效果 | 参数量 |
|------|------|--------|
| **Attention QKV** | 效果好 | 中等 |
| **Attention QKV + Output** | 效果更好 | 较多 |
| **FFN** | 额外提升 | 较多 |
| **全部** | 最好 | 最多 |

经验：先加 QKV，效果不够再加其他。

---

### Q5: LoRA 的 rank 如何选择？

**基础回答：**

rank 决定了 LoRA 的表达能力，需要根据任务复杂度和模型规模权衡选择。

**深入回答：**

**rank 影响**：

```
rank 越大:
- 表达能力越强
- 参数量越大
- 可能过拟合

rank 越小:
- 表达能力有限
- 参数量少
- 可能欠拟合
```

**经验值**：

| 模型规模 | 推荐 rank |
|----------|-----------|
| 7B 以下 | 8-16 |
| 7B-30B | 16-32 |
| 30B-70B | 32-64 |
| 70B+ | 64-128 |

**追问：如何判断 rank 是否合适？**

```python
# 方法1: 观察训练曲线
# 如果训练损失下降但验证损失上升，可能 rank 过大

# 方法2: 分析 ΔW 的秩
U, S, V = svd(ΔW)
effective_rank = sum(S > threshold) / sum(S)
# 如果 effective_rank << rank，说明 rank 过大

# 方法3: 实验对比
for r in [4, 8, 16, 32, 64]:
    model = train_lora(rank=r)
    evaluate(model)
```

**追问：LoRA 和全参数微调效果差距大吗？**

研究表明，在大多数任务上 LoRA 能达到全参数微调 95%+ 的效果：
- 简单任务：差距很小
- 复杂任务：可能有一定差距
- 新知识注入：全参数微调更好

### Q6: 除了 LoRA，还有哪些 PEFT 方法？

**基础回答：**

其他 PEFT 方法包括 Prefix Tuning、P-Tuning、Adapter、QLoRA 等。

**深入回答：**

**方法对比**：

| 方法 | 原理 | 参数量 | 优点 | 缺点 |
|------|------|--------|------|------|
| **LoRA** | 低秩分解 | 0.1-1% | 简单高效 | 表达能力有限 |
| **Prefix Tuning** | 添加可学习前缀 | 0.1% | 不改模型 | 占用序列长度 |
| **P-Tuning** | 学习 prompt embedding | <0.1% | 极轻量 | 效果不稳定 |
| **Adapter** | 层间插入小网络 | 1-5% | 模块化 | 增加推理延迟 |
| **QLoRA** | 量化 + LoRA | 极少 | 节省显存 | 推理需反量化 |

**追问：QLoRA 如何实现 4-bit 微调？**

```python
# 1. 量化基础模型
W_4bit = quantize_4bit(W_fp16)

# 2. 训练时反量化
W_compute = dequantize(W_4bit)  # 计算时用 FP16

# 3. LoRA 正常训练
output = W_compute @ x + (B @ A) @ x

# 4. 优化
- 双重量化: 量化 LoRA 参数
- 分页优化器: CPU offload 减少显存
```

**追问：如何选择 PEFT 方法？**

```
决策流程:
├── 显存受限 → QLoRA
├── 需要部署多个任务 → Adapter/LoRA
├── 追求最佳效果 → 全参数微调 或 大 rank LoRA
├── 快速实验 → LoRA (r=8-16)
└── 生产环境 → 根据资源预算选择
```

## 三、RLHF 与对齐

### Q7: 什么是 RLHF？流程是什么？

**基础回答：**

RLHF（Reinforcement Learning from Human Feedback）通过人类反馈训练奖励模型，再用强化学习优化语言模型，使模型输出更符合人类偏好。

**深入回答：**

**三阶段流程**：

```
阶段1: SFT
├── 使用高质量指令数据
└── 训练基础对话模型

阶段2: 奖励模型训练
├── 收集人类偏好数据（回答排序）
├── 训练 RM 预测人类偏好
└── 输出标量奖励分数

阶段3: PPO 训练
├── 使用 RM 作为奖励信号
├── PPO 优化语言模型策略
└── 添加 KL 散度约束防止偏离
```

**奖励模型训练**：

```python
# 给定回答 A 和 B，人类偏好 A > B
score_A = RM(response_A)
score_B = RM(response_B)

# Bradley-Terry 模型
loss = -log(sigmoid(score_A - score_B))
```

**追问：为什么需要 KL 散度约束？**

```
无约束时:
- 模型可能学会"欺骗"奖励模型
- 输出高奖励但无意义的文本
- 灾难性遗忘预训练知识

KL 约束:
loss = -reward + β × KL(π_θ || π_ref)

防止模型偏离原始模型太远
```

---

### Q8: PPO 训练的具体过程是什么？

**基础回答：**

PPO（Proximal Policy Optimization）是一种策略梯度算法，通过限制策略更新幅度来稳定训练。

**深入回答：**

**PPO 目标函数**：

```python
# 优势估计
A_t = Σ γ^l λ^l δ_{t+l}  # GAE

# PPO Clip 目标
ratio = π_θ(a|s) / π_old(a|s)
L_clip = min(ratio × A_t, clip(ratio, 1-ε, 1+ε) × A_t)

# 完整损失
L = -L_clip + c1 × L_value + c2 × L_entropy
```

**追问：PPO 中 Actor 和 Critic 分别是什么？**

```
Actor: 语言模型 π_θ
- 输入: prompt
- 输出: 生成的文本

Critic: 价值网络 V(s)
- 输入: 当前状态 (prompt + 已生成文本)
- 输出: 期望累积奖励

作用:
- Actor: 生成文本
- Critic: 评估生成质量，计算优势函数
```

**追问：RLHF 训练容易出现什么问题？**

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| **奖励欺骗** | 模型找到 RM 漏洞 | 更好的 RM 训练数据 |
| **训练不稳定** | PPO 超参数敏感 | 调参、增加约束 |
| **遗忘** | 偏离预训练模型 | 增大 KL 系数 |
| **多样性下降** | 过度优化 | 增加熵奖励 |

---

### Q9: 什么是 DPO？相比 RLHF 有什么优势？

**基础回答：**

DPO（Direct Preference Optimization）直接使用偏好数据优化模型，跳过奖励模型训练和强化学习，更简单稳定。

**深入回答：**

**DPO 原理**：

```python
# Bradley-Terry 偏好模型
P(y_w > y_l | x) = σ(r(x, y_w) - r(x, y_l))

# DPO 直接优化策略
# 推导得到最优策略与奖励的关系
π*(y|x) ∝ π_ref(y|x) × exp(r(x,y)/β)

# DPO 损失
L_DPO = -log σ(β(log π_θ(y_w|x)/π_ref(y_w|x) - log π_θ(y_l|x)/π_ref(y_l|x)))
```

**追问：DPO vs RLHF 对比？**

| 方面 | RLHF | DPO |
|------|------|-----|
| **流程复杂度** | 三阶段 | 两阶段 |
| **训练稳定性** | 需要调参 | 更稳定 |
| **计算资源** | 需要多个模型 | 只需策略模型 |
| **效果** | 理论上限更高 | 实际相当 |
| **数据需求** | 可迭代收集 | 需要成对偏好 |

**追问：DPO 有什么局限性？**

1. **需要成对偏好数据**：比单一标注更难收集
2. **难以迭代优化**：不像 RLHF 可以持续收集反馈
3. **理论上限**：无法超越奖励模型的表达能力
4. **泛化性**：对新任务可能不如 RLHF

安全对齐通过数据清洗、红队测试、对齐训练等方式减少模型的有害输出。

## 四、量化技术

### Q11: 什么是模型量化？为什么需要量化？

**基础回答：**

量化是将模型参数从高精度（如 FP16）转换为低精度（如 INT8、INT4），减少模型大小和计算量。

**深入回答：**

**量化类型**：

| 类型 | 方法 | 精度损失 | 适用场景 |
|------|------|----------|----------|
| **训练后量化 (PTQ)** | 训练后直接量化 | 较大 | 部署阶段 |
| **量化感知训练 (QAT)** | 训练时模拟量化 | 较小 | 训练阶段 |
| **混合精度** | 部分层量化 | 中等 | 平衡选择 |

**精度对比**：

```
FP16: 16 位浮点
  - 范围: ±65504
  - 精度: ~0.001

INT8: 8 位整数
  - 范围: -128 ~ 127
  - 需要 scale 和 zero_point

INT4: 4 位整数
  - 范围: -8 ~ 7
  - 精度损失更明显
```

**追问：量化如何实现？**

```python
# 线性量化公式
q = round(r / scale) + zero_point
r = scale × (q - zero_point)

# 其中:
# r: 原始浮点值
# q: 量化整数值
# scale: 缩放因子
# zero_point: 零点偏移

# INT8 量化示例
scale = (max_val - min_val) / 255
zero_point = round(-min_val / scale)
q = round(x / scale) + zero_point
```

**追问：量化对模型效果影响多大？**

| 量化精度 | 模型大小 | 推理速度 | 效果损失 |
|----------|----------|----------|----------|
| FP16 | 100% | 基准 | 0% |
| INT8 | 50% | 2-4x | <1% |
| INT4 | 25% | 更快 | 1-3% |
| INT3 | 19% | 更快 | 3-5% |
| INT2 | 12.5% | 更快 | >5% |

### Q12: GPTQ 和 AWQ 量化有什么区别？

**基础回答：**

GPTQ 和 AWQ 都是常用的训练后量化方法，但实现思路不同。

**深入回答：**

**GPTQ 原理**：

```
基于 Optimal Brain Quantization:
1. 逐层量化
2. 使用校准数据计算 Hessian 矩阵
3. 最小化量化误差的期望
4. 逐个权重更新以补偿误差
```

**AWQ 原理**：

```
Activation-aware Weight Quantization:
1. 分析权重的重要性（基于激活值）
2. 保护重要权重（保持高精度）
3. 对不重要权重量化
4. 缩放因子优化
```

**追问：两者的优缺点对比？**

| 方面 | GPTQ | AWQ |
|------|------|-----|
| **量化速度** | 较慢（需校准） | 较快 |
| **推理效果** | 略好 | 相当 |
| **显存占用** | 量化时较高 | 较低 |
| **实现复杂度** | 中等 | 较低 |
| **适用场景** | 追求最佳效果 | 快速部署 |

**追问：如何选择量化方法？**

```
决策流程:
├── 追求最佳效果 → GPTQ (或 AWQ)
├── 快速部署 → AWQ
├── 极低显存 → QLoRA 训练方式
├── 需要 fine-tuning → QAT 或 QLoRA
└── 生产环境 → 测试后选择
```

## 五、推理优化

### Q13: 如何优化大模型推理速度？

**基础回答：**

推理优化方法包括 KV Cache、模型量化、推测解码、并行推理等。

**深入回答：**

**优化方法分类**：

```
┌─────────────────────────────────────────────┐
│              推理优化技术                     │
├─────────────────────────────────────────────┤
│                                             │
│  计算优化                                    │
│  ├── FlashAttention                         │
│  ├── FlashDecoding                          │
│  ├── 算子融合                               │
│  └── KV Cache                               │
│                                             │
│  模型优化                                    │

│  └── 知识蒸馏                               │
│                                             │
│  系统优化                                    │
│  ├── 批处理优化                             │
│  ├── 分布式推理                             │
│  ├── 推测解码                               │
│  └── Continuous Batching                    │
│                                             │
└─────────────────────────────────────────────┘
```

**追问：Continuous Batching 是什么？**

```
传统 Batching:
- 等待所有请求完成才开始下一批
- 短请求等待长请求，浪费计算

Continuous Batching:
- 请求完成后立即释放资源
- 新请求可以随时加入
- 显著提高吞吐量
```

**追问：vLLM 如何实现高效推理？**

vLLM 的核心创新：**PagedAttention**

```
传统 KV Cache:
- 预分配最大长度内存
- 内存碎片、浪费

PagedAttention:
- 将 KV Cache 分成固定大小的 block
- 按需分配，类似操作系统的虚拟内存
- 支持内存共享（beam search 等）
```

### Q14: KV Cache 的原理和优化方法？

**基础回答：**

KV Cache 缓存之前计算的 Key 和 Value，避免重复计算，加速自回归生成。

**深入回答：**

**工作原理**：

```python
# 第 t 步生成
# 无 Cache: 需要重新计算所有位置的 K, V
for i in range(t+1):
    K[i], V[i] = compute_kv(x[i])
output = attention(Q[t], K[:t+1], V[:t+1])

# 有 Cache: 只计算当前位置
K[t], V[t] = compute_kv(x[t])  # 只计算新位置
output = attention(Q[t], cached_K + K[t], cached_V + V[t])
```

**内存占用计算**：

```
KV Cache 大小 = 2 × n_layers × n_heads × head_dim × seq_len × bytes_per_element

例如 LLaMA 7B (seq_len=4096, FP16):
= 2 × 32 × 32 × 128 × 4096 × 2
= 2 GB
```

**追问：如何减少 KV Cache 内存？**

| 方法 | 原理 | 效果 |
|------|------|------|
| **MQA/GQA** | 共享 KV | 减少 4-8 倍 |
| **KV Cache 量化** | INT8/INT4 | 减少 2-4 倍 |
| **PagedAttention** | 按需分配 | 减少碎片 |
| **滑动窗口** | 只保留最近 | 减少到固定大小 |
| **多查询注意力** | 批量共享 | 提高效率 |

### Q15: 如何实现大模型的分布式推理？

**基础回答：**

分布式推理将模型分片到多个 GPU 上，支持大模型部署和高吞吐推理。

**深入回答：**

**并行策略**：

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| **张量并行 (TP)** | 按层内切分 | 单机多卡 |
| **流水线并行 (PP)** | 按层间切分 | 多机部署 |
| **序列并行 (SP)** | 按序列切分 | 长序列推理 |

**张量并行示意**：

```python
# 矩阵乘法切分
# Y = XW = X[W₁ | W₂] = [XW₁ | XW₂]

# Attention 层切分
# 每个 GPU 处理部分 head
# GPU 1: head 0-15
# GPU 2: head 16-31

# 通信点:
# - Attention 输出后的 All-Reduce
# - FFN 后的 All-Reduce
```

**追问：如何选择并行策略？**

```
决策因素:
├── 模型大小
│   ├── 单卡能放下 → 单卡推理
│   ├── 需要多卡 → 张量并行
│   └── 需要多机 → TP + PP
│
├── 序列长度
│   └── 超长序列 → 加入序列并行
│
├── 批量大小
│   └── 大批量 → 数据并行 + TP
│
└── 延迟要求
    └── 低延迟 → TP（通信快）
```

### Q16: 微调时遇到 Loss 震荡怎么办？

**基础回答：**

Loss 震荡可能由学习率过大、数据问题、梯度异常等原因导致。

**深入回答：**

**排查步骤**：

```
1. 检查学习率
   ├── 是否过大（尝试减小）
   └── 是否需要 warmup

2. 检查数据
   ├── 是否有异常样本
   ├── 标签是否正确
   └── 数据分布是否均衡

3. 检查梯度
   ├── 是否有梯度爆炸（添加裁剪）
   └── 是否有 NaN/Inf

4. 检查模型
   ├── 是否有层未正确加载
   └── 是否有参数未正确冻结
```

**追问：如何调整学习率？**

```python
# 方法1: 降低学习率

# 方法2: 增加 warmup
warmup_steps = total_steps * 0.1  # 10% warmup

# 方法3: 使用更平滑的调度器
```
