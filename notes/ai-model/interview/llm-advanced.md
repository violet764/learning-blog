# 大模型核心原理面试题

本章节整理了大语言模型核心原理相关的面试题目，涵盖预训练、涌现能力、上下文学习、推理能力等主题。

---

## 一、预训练原理

### Q1: 大模型预训练的目标是什么？

**基础回答：**

大模型预训练的目标是通过海量无标注文本学习语言的统计规律和语义知识，获得通用的语言理解和生成能力。

**深入回答：**

**预训练目标对比**：

| 目标 | 公式 | 特点 | 代表模型 |
|------|------|------|----------|
| **自回归 (CLM)** | P(x) = ∏P(x_t\|x_{<t}) | 单向，生成式 | GPT、LLaMA |
| **掩码语言模型 (MLM)** | P(x_mask\|x_visible) | 双向，理解式 | BERT |
| **Prefix LM** | 前缀双向 + 后缀单向 | 结合两者 | T5、PaLM |

**追问：为什么 GPT 选择自回归而不是 MLM？**

1. **生成能力**：自回归天然支持文本生成
2. **概率建模**：直接建模 P(x)，可用于评估、采样
3. **通用性**：一个模型可以处理多种生成任务
4. **工程简单**：无需设计 mask 策略

**追问：预训练数据的规模如何影响模型能力？**

```
Scaling Laws: L(N, D) = A/N^α + B/D^β + E

其中:
- N: 参数量
- D: 数据量
- L: 损失
- α, β, A, B, E: 拟合参数

结论: 模型性能与参数量和数据量的幂次成正比
```

### Q2: 预训练数据处理流程是什么？

**基础回答：**

预训练数据处理包括数据收集、清洗、去重、分词等步骤。

**深入回答：**

**完整流程**：

```
1. 数据收集
   ├── 网页爬取 (Common Crawl)
   ├── 书籍 (Books3, Gutenberg)
   ├── 代码 (GitHub)
   ├── 维基百科
   └── 学术论文 (arXiv)

2. 数据清洗
   ├── 去除 HTML 标签
   ├── 过滤低质量内容
   ├── 去除敏感信息
   └── 语言过滤

3. 数据去重
   ├── 精确去重
   ├── 模糊去重 (MinHash)
   └── 文档级/句子级去重

4. 数据混合
   ├── 不同来源配比
   ├── 领域平衡
   └── 质量加权

5. 分词
   ├── 训练分词器
   └── 编码文本
```

**追问：为什么需要去重？**

1. **防止记忆**：重复数据让模型记住而非学习
2. **提高效率**：减少无效计算
3. **公平评估**：避免测试集泄露

**追问：LLaMA 的数据配比是怎样的？**

```
LLaMA 训练数据配比:
- CommonCrawl: 67%
- C4: 15%
- GitHub: 4.5%
- Wikipedia: 4.5%
- Books: 4.5%
- ArXiv: 2.5%
- StackExchange: 2%
```

### Q3: 什么是 Scaling Laws？它对大模型训练有什么指导意义？

**基础回答：**

Scaling Laws 描述了模型性能与计算量、参数量、数据量之间的幂律关系。

**深入回答：**

**核心发现**（Kaplan et al., 2020）：

```
L(N) = (N_c/N)^α_N        # 损失与参数量
L(D) = (D_c/D)^α_D        # 损失与数据量
L(C) = (C_c/C)^α_C        # 损失与计算量

其中 α_N ≈ 0.076, α_D ≈ 0.095, α_C ≈ 0.050
```

**Chinchilla 定律**（Hoffmann et al., 2022）：

```
给定计算预算 C，最优参数量和数据量:
N_opt ∝ C^0.5
D_opt ∝ C^0.5

结论: 参数量和数据量应该等比例缩放
```

**追问：LLaMA 如何应用 Scaling Laws？**

LLaMA 选择"过度训练"（train more than Chinchilla optimal）：
- Chinchilla: 70B 模型训练 1.4T tokens
- LLaMA: 65B 模型训练 2T tokens

原因：
1. 推理成本更重要（模型训练一次，推理多次）
2. 小模型推理更快
3. 更好的性能/推理成本比

**追问：Scaling Laws 有什么局限性？**

1. **不适用于所有任务**：某些能力（如推理）可能有不同的缩放规律
2. **数据质量未考虑**：只考虑数量，不考虑质量
3. **架构无关假设**：不同架构可能有不同的缩放规律
4. **外推不确定性**：超出观测范围时预测不准确

## 二、涌现能力

### Q4: 什么是涌现能力（Emergent Abilities）？

**基础回答：**

涌现能力是指模型在达到一定规模后突然出现的能力，这些能力在小模型中几乎不存在。

**深入回答：**

**典型涌现能力**：

| 能力 | 描述 | 涌现规模 |
|------|------|----------|
| **Few-shot Learning** | 少量示例学习新任务 | ~10B |
| **Chain-of-Thought** | 逐步推理 | ~100B |
| **指令遵循** | 理解并执行复杂指令 | ~50B |
| **代码生成** | 生成可执行代码 | ~10B |
| **数学推理** | 解决数学问题 | ~100B |

**追问：涌现能力为什么会出现？**

几种假说：

1. **相变假说**：类似物理相变，超过临界点后性质突变
2. **组合假说**：模型学会了组合基础能力
3. **评估假说**：与评估指标有关（如精确匹配更容易"涌现"）

**追问：涌现能力真的存在吗？**

近期研究（Schaeffer et al., 2023）质疑：
- 可能是评估指标导致的"幻觉"
- 使用平滑指标后，涌现现象消失
- 能力可能是渐进提升的

### Q5: 什么是上下文学习（In-Context Learning）？

**基础回答：**

上下文学习是指模型通过在输入中提供示例，无需参数更新就能学习新任务的能力。

**深入回答：**

**形式化描述**：

```
输入格式:
任务描述 + 示例1 + 示例2 + ... + 示例n + 查询

模型输出:
基于示例推断任务并完成查询
```

**示例**：

```
将英文翻译为中文:
English: Hello
Chinese: 你好
English: Good morning
Chinese: 早上好
English: Thank you
Chinese: ?
```

**追问：In-Context Learning 的原理是什么？**

几种理论解释：

1. **隐式微调假说**：
   - 前向传播中的注意力机制类似于梯度下降
   - 每层可以看作一次"更新"

2. **贝叶斯推断假说**：
   - 模型学习到了任务的先验分布
   - In-Context 相当于后验推断

3. **模式匹配假说**：
   - 预训练中见过类似模式
   - 直接匹配并复用

**追问：如何提升 In-Context Learning 效果？**

1. **示例选择**：选择与查询相似的示例
2. **示例顺序**：相关示例放后面效果更好
3. **示例数量**：通常 3-8 个效果较好
4. **格式统一**：保持示例格式一致

### Q6: 什么是 Chain-of-Thought（思维链）？

**基础回答：**

Chain-of-Thought（CoT）是一种提示技术，让模型逐步推理得出答案，而不是直接输出结果。

**深入回答：**

**对比**：

```
标准提示:
Q: 小明有 5 个苹果，给了小红 2 个，又买了 3 个，现在有几个？
A: 6

CoT 提示:
Q: 小明有 5 个苹果，给了小红 2 个，又买了 3 个，现在有几个？
A: 让我一步步思考。
   1. 小明原来有 5 个苹果
   2. 给了小红 2 个后，还剩 5 - 2 = 3 个
   3. 又买了 3 个，现在有 3 + 3 = 6 个
   答案是 6。
```

**追问：CoT 为什么有效？**

1. **分解问题**：将复杂问题分解为简单步骤
2. **激活知识**：逐步推理激活相关预训练知识
3. **增加计算**：更多 token 意味着更多计算
4. **可解释性**：推理过程可见，便于调试

**追问：CoT 的变体有哪些？**

| 变体 | 描述 |
|------|------|
| **Zero-shot CoT** | 添加"让我们一步步思考" |
| **Few-shot CoT** | 提供带推理步骤的示例 |
| **Self-Consistency** | 多次采样，投票选择答案 |
| **Tree of Thoughts** | 构建推理树，搜索最优路径 |
| **Auto-CoT** | 自动生成推理步骤 |

## 三、模型架构

### Q7: 为什么主流 LLM 都采用 Decoder-only 架构？

**基础回答：**

Decoder-only 架构实现简单、训练效率高、生成能力强，已成为主流选择。

**深入回答：**

**三种架构对比**：

| 架构 | 代表模型 | 优点 | 缺点 |
|------|----------|------|------|
| **Encoder-only** | BERT | 双向理解强 | 不适合生成 |
| **Decoder-only** | GPT, LLaMA | 生成能力强，实现简单 | 单向理解 |
| **Encoder-Decoder** | T5, BART | 理解+生成平衡 | 实现复杂 |

**追问：为什么 Decoder-only 比 Encoder-Decoder 更流行？**

1. **实现简单**：只需要一个单向 Attention
2. **扩展性好**：容易扩展到更大规模
3. **In-Context Learning**：更适合上下文学习
4. **工程成熟**：生态更完善

**追问：Decoder-only 的单向注意力有什么问题？**

- 无法看到后续上下文（某些理解任务不利）
- 通过 In-Context Learning 部分弥补
- 可以通过多种技巧缓解（如 Prefix LM）

---

### Q8: LLaMA 相比 GPT 有哪些改进？

**基础回答：**

LLaMA 在架构上做了多处优化，包括 RMSNorm、SwiGLU、RoPE 等。

**深入回答：**

**架构对比**：

| 组件 | GPT-3 | LLaMA |
|------|-------|-------|
| **归一化** | LayerNorm (Post-Norm) | RMSNorm (Pre-Norm) |
| **激活函数** | GELU | SwiGLU |
| **位置编码** | 正弦位置编码 | RoPE |
| **FFN 维度** | 4 × d_model | (8/3) × d_model |
| **Attention** | MHA | GQA (LLaMA 2/3) |

**追问：RMSNorm 和 LayerNorm 的区别？**

```python
# LayerNorm
y = (x - mean) / sqrt(var + ε) * γ + β

# RMSNorm (去掉均值归一化)
y = x / sqrt(mean(x²) + ε) * γ
```

RMSNorm 去掉了均值归一化和偏置项 β：
- 计算更简单
- 效果相当或略好
- 训练更稳定

**追问：SwiGLU 相比 GELU 的优势？**

```python
# GELU FFN
FFN(x) = GELU(xW₁)W₂

# SwiGLU FFN
FFN(x) = (Swish(xW₁) ⊙ xV)W₂
```

SwiGLU：
- 引入门控机制，表达能力更强
- 参数量增加约 50%（需要额外 V 矩阵）
- 实验表明效果更好

### Q9: 什么是混合专家模型（MoE）？

**基础回答：**

MoE（Mixture of Experts）将 FFN 层替换为多个专家网络，通过门控机制选择激活部分专家。

**深入回答：**

**架构示意**：

```
输入 x
    ↓
┌─────────────────────────┐
│      Gate Network       │ → 选择 Top-k 专家
└─────────────────────────┘
    ↓
┌────────┬────────┬────────┬────────┐
│ Expert1│ Expert2│ Expert3│ Expert4│
└────────┴────────┴────────┴────────┘
    ↓         ↓
  加权组合输出
```

**计算过程**：

```python
# 门控选择
gates = softmax(Gate(x))  # (n_experts,)
topk_gates, topk_indices = topk(gates, k)

# 专家计算
expert_outputs = [Experts[i](x) for i in topk_indices]

# 加权组合
output = sum(gate * out for gate, out in zip(topk_gates, expert_outputs))
```

**追问：MoE 的优势是什么？**

| 优势 | 说明 |
|------|------|
| **参数效率** | 参数量大但计算量可控 |
| **专业化** | 不同专家学习不同知识 |
| **可扩展** | 容易扩展专家数量 |

**追问：MoE 有什么挑战？**

1. **负载均衡**：某些专家可能很少被激活
2. **训练不稳定**：门控可能坍缩到少数专家
3. **通信开销**：分布式训练时专家间通信
4. **推理优化**：需要处理动态路由

**追问：如何解决负载均衡问题？**

```python
# 添加辅助损失
aux_loss = α × ∑f_i × P_i

其中:
- f_i: 专家 i 被选择的频率
- P_i: 专家 i 的平均门控值
- α: 平衡系数

这鼓励所有专家都被均匀使用
```

## 四、推理与生成

### Q10: 大模型生成文本的解码策略有哪些？

**基础回答：**

常用解码策略包括贪婪搜索、Beam Search、Top-k、Top-p（核采样）、温度调节等。

**深入回答：**

**策略对比**：

| 策略 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **贪婪** | 选择概率最高的词 | 简单快速 | 容易重复、缺乏多样性 |
| **Beam Search** | 保留 top-k 候选序列 | 平衡质量与多样性 | 可能生成通用内容 |
| **Top-k** | 从概率最高的 k 个词中采样 | 增加多样性 | k 值选择困难 |
| **Top-p** | 从累积概率达 p 的词中采样 | 自适应范围 | 可能选择概率低的词 |
| **温度** | 调整概率分布平滑度 | 控制随机性 | 极端值效果差 |

**公式**：

```python
# 温度调整
prob = softmax(logits / temperature)

# temperature → 0: 趋近贪婪
# temperature → ∞: 趋近均匀采样
# temperature = 1: 原始分布
```

**追问：为什么大模型常用 Top-p 而不是 Beam Search？**

1. **多样性更好**：避免生成"安全"但无聊的内容
2. **效率更高**：不需要维护多个候选序列
3. **质量足够**：大模型本身生成质量已很高
4. **自然感更强**：更接近人类写作的多样性

**追问：如何减少生成重复？**

1. **重复惩罚**：`logit = logit - penalty × count(recent_tokens)`
2. **n-gram 惩罚**：禁止生成已出现的 n-gram
3. **调整温度**：适当提高温度增加随机性
4. **截断采样**：结合 Top-p 和 Top-k

### Q11: 什么是推测解码（Speculative Decoding）？

**基础回答：**

推测解码使用一个小模型快速生成候选序列，然后用大模型并行验证，加速推理。

**深入回答：**

**流程**：

```
1. 小模型 (Draft Model) 自回归生成 k 个候选 token

2. 大模型 (Target Model) 一次性验证所有候选:
   - 并行计算 k 个位置的 logits
   - 比较小模型和大模型的概率分布
   - 接受或拒绝每个 token

3. 如果某 token 被拒绝，从正确分布采样替代
```

**追问：为什么能加速？**

```
标准生成: 每生成 1 个 token 需要 1 次大模型前向

推测解码: 
- 假设小模型生成 k 个 token
- 大模型验证: 1 次前向（并行）
- 假设平均接受 k' 个 token
- 加速比: k' / (1 + k_small_forward)
```

当小模型和大模型概率分布接近时，加速效果明显。

**追问：小模型如何选择？**

1. **同系列更小版本**：如 LLaMA 70B + LLaMA 7B
2. **蒸馏模型**：专门蒸馏的草稿模型
3. **自推测解码**：模型自身的早期层作为草稿

### Q12: 如何评估大模型的能力？

**基础回答：**

大模型评估包括基准测试、人类评估、模型评估等多种方法。

**深入回答：**

**评估维度**：

| 维度 | 常用基准 | 说明 |
|------|----------|------|
| **知识能力** | MMLU, C-Eval | 多学科知识 |
| **推理能力** | GSM8K, MATH | 数学推理 |
| **代码能力** | HumanEval, MBPP | 代码生成 |
| **语言理解** | HellaSwag, PIQA | 常识推理 |
| **长文本** | LongBench, L-Eval | 长上下文理解 |
| **安全性** | TruthfulQA, CValues | 安全与价值观 |

**追问：自动化评估有什么问题？**

1. **数据泄露**：测试数据可能出现在训练集中
2. **评估偏见**：基准可能偏向某些模型
3. **能力覆盖不全**：难以覆盖所有能力
4. **与人类偏好不一致**：高分不等于好用

**追问：什么是 LLM-as-Judge？**

使用强大的 LLM（如 GPT-4）评估其他模型的输出：

```
评估流程:
1. 给定问题和多个模型的回答
2. 让评估模型打分或排序
3. 汇总得到模型排名

优点: 可扩展、一致性高
缺点: 偏见、成本高、可能偏向特定风格
```

## 五、高级问题

### Q13: 什么是长度外推？如何实现？

**基础回答：**

长度外推是指模型处理比训练时更长的序列的能力。

**深入回答：**

**问题分析**：

```
训练时最大长度: 2048
推理时需要长度: 8000

问题:
1. 位置编码超出训练范围
2. 注意力模式不适应更长序列
3. 内存可能不足
```

**解决方案**：

| 方法 | 原理 | 效果 |
|------|------|------|
| **位置插值 (PI)** | 将长序列位置压缩到训练范围内 | 简单有效 |
| **NTK-aware 插值** | 根据频率调整插值比例 | 更好保留高频信息 |
| **YaRN** | 结合温度缩放和 NTK | 当前最优 |
| **ALiBi** | 线性偏置天然支持外推 | 不需要训练 |

**追问：RoPE 如何实现长度外推？**

```python
# 位置插值 (Position Interpolation)
# 原始位置: 0, 1, 2, ..., L'
# 压缩位置: 0, L/L', 2L/L', ..., L

def position_interpolate(pos, max_train_len, max_new_len):
    scale = max_train_len / max_new_len
    return pos * scale
```

### Q14: 大模型为什么会产生幻觉？如何减少？

**基础回答：**

幻觉是指模型生成与事实不符或无法验证的内容。产生原因包括训练数据问题、解码策略等。

**深入回答：**

**幻觉类型**：

| 类型 | 描述 | 示例 |
|------|------|------|
| **事实性幻觉** | 与已知事实矛盾 | "太阳从西边升起" |
| **忠实性幻觉** | 与输入上下文矛盾 | 输入说 A，输出说非 A |
| **无根据幻觉** | 无法验证的内容 | 编造不存在的论文 |

**产生原因**：

1. **训练数据噪声**：数据本身存在错误
2. **曝光偏差**：训练和推理分布不一致
3. **概率采样**：低概率词也可能被采样
4. **过度自信**：模型对错误答案也给出高概率

**追问：如何减少幻觉？**
