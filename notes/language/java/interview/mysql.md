## 一、索引

### 1. MySQL 索引的数据结构？

**答案：**

MySQL 使用 **B+树** 作为索引结构。

**B+树特点：**

```
                    ┌─────────────┐
                    │   根节点     │
                    │  只存索引    │
                    └──────┬──────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │内部节点 │  │内部节点 │  │内部节点 │
        │只存索引 │  │只存索引 │  │只存索引 │
        └────┬────┘  └────┬────┘  └────┬────┘
             │            │            │
        ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
        ▼         ▼  ▼         ▼  ▼         ▼
    ┌───────┐┌───────┐┌───────┐┌───────┐┌───────┐
    │叶子节点││叶子节点││叶子节点││叶子节点││叶子节点│
    │存数据  ││存数据  ││存数据  ││存数据  ││存数据  │
    └───┬───┘└───┬───┘└───┬───┘└───┬───┘└───┬───┘
        └────────┴────────┴────────┴────────┘
                    指针连接（便于范围查询）
```

**为什么用 B+树？**

| 结构 | 问题 |
|------|------|
| B 树 | 非叶子存数据，节点存索引少，树更高 |
| Hash | 不支持范围查询 |
| 二叉树/红黑树 | 树太高，磁盘 IO 多 |
| B+树 | 非叶子只存索引，树更矮，磁盘 IO 少；叶子有序，支持范围查询 |

### 2. 聚簇索引和非聚簇索引？

**答案：**

| 类型 | 说明 | 存储位置 |
|------|------|----------|
| 聚簇索引 | 主键索引，叶子存完整数据 | 数据文件 |
| 非聚簇索引 | 二级索引，叶子存主键值 | 索引文件 |

```
聚簇索引（主键索引）：
┌─────────────────────────────────────┐
│  叶子节点：完整的一行数据              │
│  id=1, name='张三', age=25          │
└─────────────────────────────────────┘

非聚簇索引（二级索引）：
┌─────────────────────────────────────┐
│  叶子节点：索引列值 + 主键值           │
│  name='张三' → id=1                 │
└─────────────────────────────────────┘
```

**追问：什么是回表？**

> 通过二级索引查到主键值，再回主键索引查数据。
> 
> ```sql
> -- name 上有索引
> SELECT * FROM user WHERE name = '张三';
> -- 1. 先查 name 索引，得到 id=1
> -- 2. 再回 id 索引查完整数据（回表）
> ```
> 
> 避免回表：使用覆盖索引
> ```sql
> SELECT id, name FROM user WHERE name = '张三';
> -- name 索引已包含 id 和 name，无需回表
> ```

### 3. 索引什么情况下会失效？

**答案：**

```sql
-- 1. 索引列参与计算
SELECT * FROM user WHERE id + 1 = 10;

-- 2. 索引列使用函数
SELECT * FROM user WHERE UPPER(name) = 'ZHANG';

-- 3. 类型隐式转换
-- name 是 varchar
SELECT * FROM user WHERE name = 123;  -- 隐式转为字符串

-- 4. LIKE 以 % 开头
SELECT * FROM user WHERE name LIKE '%张';

-- 5. 使用 != 或 <>
SELECT * FROM user WHERE status != 1;

-- 6. 使用 OR（除非所有字段都有索引）
SELECT * FROM user WHERE name = '张三' OR age = 25;

-- 7. 索引列允许 NULL 且查询 IS NULL
-- （取决于优化器判断）

-- 8. 复合索引不满足最左匹配
-- 索引：(name, age)
SELECT * FROM user WHERE age = 25;  -- 失效
SELECT * FROM user WHERE name = '张三';  -- 生效
SELECT * FROM user WHERE name = '张三' AND age = 25;  -- 生效
```

### 4. 最左匹配原则？

**答案：**

复合索引按定义顺序使用，从最左列开始匹配。

```sql
-- 索引：(a, b, c)

-- 命中索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- 不命中索引
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3

-- 部分命中
WHERE a = 1 AND c = 3  -- 只命中 a
```

**原理：**

```
索引结构按列顺序组织：

(a=1, b=1, c=1) → (a=1, b=1, c=2) → (a=1, b=2, c=1) → (a=2, b=1, c=1)
                                                       ↑
                                                       a 不同，有序
                        ↑
                        b 在 a=1 范围内有序

跳过 a 直接查 b：
无法利用有序性，因为不同 a 下 b 无序
```

## 二、事务

### 5. 事务的 ACID 特性？

**答案：**

| 特性 | 说明 |
|------|------|
| 原子性（Atomicity） | 事务是不可分割的最小单位 |
| 一致性（Consistency） | 事务执行前后数据保持一致状态 |
| 隔离性（Isolation） | 多个事务并发执行互不干扰 |
| 持久性（Durability） | 事务提交后数据永久保存 |

**追问：MySQL 如何实现 ACID？**

> - **原子性**：undo log（回滚日志）
> - **一致性**：数据库约束 + 业务逻辑
> - **隔离性**：锁 + MVCC
> - **持久性**：redo log（重做日志）

---

### 6. 事务的隔离级别？

**答案：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|:----:|:----------:|:----:|
| 读未提交（READ UNCOMMITTED） | ✓ | ✓ | ✓ |
| 读已提交（READ COMMITTED） | ✗ | ✓ | ✓ |
| 可重复读（REPEATABLE READ） | ✗ | ✗ | ✓ |
| 串行化（SERIALIZABLE） | ✗ | ✗ | ✗ |

```
脏读：读到其他事务未提交的数据
不可重复读：同一事务两次读取结果不同（修改导致）
幻读：同一事务两次读取记录数不同（插入/删除导致）
```

**MySQL 默认：可重复读**

**追问：MySQL 如何解决幻读？**

> 1. **MVCC**：快照读，看到事务开始时的数据版本
> 2. **Next-Key Lock**：当前读，锁定记录和间隙，防止插入

---

### 7. MVCC 是什么？

**答案：**

MVCC（多版本并发控制）通过数据版本来实现并发控制。

```
每行数据包含：
├── DB_TRX_ID：最后修改该行的事务 ID
├── DB_ROLL_PTR：回滚指针，指向 undo log
└── DB_ROW_ID：隐藏主键（无主键时）

Read View（读视图）：
├── m_ids：活跃事务 ID 列表
├── min_trx_id：最小活跃事务 ID
├── max_trx_id：下一个事务 ID
└── creator_trx_id：创建该 Read View 的事务 ID

可见性判断：
├── DB_TRX_ID < min_trx_id → 可见（事务已提交）
├── DB_TRX_ID >= max_trx_id → 不可见（事务在 Read View 后开启）
├── DB_TRX_ID in m_ids → 不可见（事务未提交）
└── 其他 → 可见
```

## 三、锁

### 8. MySQL 有哪些锁？

**答案：**

```
按粒度分：
├── 全局锁：锁定整个数据库（备份用）
├── 表级锁
│   ├── 表锁
│   ├── 元数据锁（MDL）
│   └── 意向锁
└── 行级锁
    ├── 记录锁（Record Lock）
    ├── 间隙锁（Gap Lock）
    └── 临键锁（Next-Key Lock）

按类型分：
├── 共享锁（S）：读锁
└── 排他锁（X）：写锁
```

**追问：什么时候用行锁，什么时候用表锁？**

> - **行锁**：InnoDB，通过索引访问数据时使用
> - **表锁**：无索引访问、MyISAM、显式 LOCK TABLES

---

### 9. Next-Key Lock 是什么？

**答案：**

Next-Key Lock = Record Lock + Gap Lock

```
记录锁：锁定索引记录本身
间隙锁：锁定记录之间的间隙

示例：表中有 id = 1, 5, 10

SELECT * FROM user WHERE id = 5 FOR UPDATE;

Next-Key Lock 锁定：
├── (1, 5]：间隙 + 记录
└── (5, 10]：间隙 + 记录

防止：
├── 其他事务插入 id = 3（间隙锁阻止）
└── 其他事务修改 id = 5（记录锁阻止）
```

**追问：什么时候退化为间隙锁？**

> 唯一索引等值查询，记录存在时：
> - 只加记录锁，不加间隙锁
>
> 非唯一索引等值查询：
> - 加 Next-Key Lock + 间隙锁

## 四、SQL 优化

### 10. 如何进行 SQL 优化？

**答案：**

```
1. 分析执行计划
   EXPLAIN SELECT ...

2. 索引优化
   ├── 避免 SELECT *
   ├── WHERE、ORDER BY、GROUP BY 字段建索引
   ├── 使用覆盖索引避免回表
   └── 避免索引失效情况

3. 查询优化
   ├── 避免 SELECT *
   ├── 小表驱动大表
   ├── 用 EXISTS 替代 IN
   └── 避免子查询，用 JOIN

4. 表结构优化
   ├── 选择合适的字段类型
   ├── 适当的反范式（减少 JOIN）
   ├── 垂直拆分（大表拆小表）
   └── 水平拆分（分库分表）
```

---

### 11. EXPLAIN 重要字段？

**答案：**

| 字段 | 说明 |
|------|------|
| id | 执行顺序，越大越先执行 |
| type | 访问类型，从好到差：system > const > eq_ref > ref > range > index > ALL |
| key | 实际使用的索引 |
| rows | 预估扫描行数 |
| Extra | 额外信息 |

**Extra 重要值：**

```
Using index：覆盖索引，不需要回表
Using where：WHERE 过滤
Using filesort：文件排序（需优化）
Using temporary：使用临时表（需优化）
```

## 五、其他问题

### 12. COUNT(*) 和 COUNT(1) 和 COUNT(字段)？

**答案：**

```sql
-- COUNT(*) 和 COUNT(1)：统计所有行
-- 性能相同，MySQL 优化后一致
SELECT COUNT(*) FROM user;
SELECT COUNT(1) FROM user;

-- COUNT(字段)：统计该字段非 NULL 的行数
SELECT COUNT(name) FROM user;

-- 性能：COUNT(*) ≈ COUNT(1) > COUNT(字段)
```

---

### 13. CHAR 和 VARCHAR 的区别？

**答案：**

| 类型 | 存储 | 最大长度 | 适用场景 |
|------|------|----------|----------|
| CHAR | 定长 | 255 | 长度固定（手机号、MD5） |
| VARCHAR | 变长 | 65535 | 长度不固定（用户名、地址） |

```sql
CHAR(10)：存储 'abc'，实际占 10 字节
VARCHAR(10)：存储 'abc'，实际占 4 字节（3 + 1 长度）
```

---

### 14. MyISAM 和 InnoDB 的区别？

**答案：**

| 特性 | MyISAM | InnoDB |
|------|--------|--------|
| 事务 | 不支持 | 支持 |
| 锁 | 表锁 | 行锁 + 表锁 |
| 外键 | 不支持 | 支持 |
| 索引 | 非聚簇 | 聚簇 |
| 崩溃恢复 | 不支持 | 支持 |
| 全文索引 | 支持 | 5.6+ 支持 |
| 推荐程度 | 不推荐 | 推荐 |
