# JVM 面试题

> 本文档整理 JVM（Java 虚拟机）相关的高频面试题。

## 一、内存模型

### 1. JVM 内存模型是怎样的？

**答案：**

```
┌─────────────────────────────────────────────────────┐
│                  JVM 运行时数据区                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌─────────────────┐  ┌─────────────────────────┐  │
│  │     方法区       │  │          堆             │  │
│  │  (Method Area)  │  │        (Heap)          │  │
│  │                 │  │                         │  │
│  │ 类信息、常量、   │  │   对象实例、数组        │  │
│  │ 静态变量        │  │                         │  │
│  └─────────────────┘  └─────────────────────────┘  │
│        ↓ 线程共享              ↓ 线程共享           │
├─────────────────────────────────────────────────────┤
│  ┌─────────────────┐  ┌─────────────────────────┐  │
│  │    虚拟机栈      │  │       本地方法栈        │  │
│  │   (VM Stack)    │  │ (Native Method Stack)  │  │
│  │                 │  │                         │  │
│  │ 栈帧、局部变量、 │  │     Native 方法        │  │
│  │ 操作数栈        │  │                         │  │
│  └─────────────────┘  └─────────────────────────┘  │
│        ↓ 线程私有              ↓ 线程私有           │
├─────────────────────────────────────────────────────┤
│                                                     │
│              程序计数器 (PC Register)               │
│                                                     │
│              ↓ 线程私有                             │
└─────────────────────────────────────────────────────┘
```

| 区域 | 作用 | 线程安全 | 异常 |
|------|------|:--------:|------|
| 堆 | 存储对象实例 | 不安全 | OutOfMemoryError |
| 方法区 | 类信息、常量 | 不安全 | OutOfMemoryError |
| 虚拟机栈 | 方法调用、局部变量 | 安全 | StackOverflowError、OOM |
| 本地方法栈 | Native 方法 | 安全 | StackOverflowError、OOM |
| 程序计数器 | 记录执行位置 | 安全 | 无 |

### 2. 堆的内存结构？

**答案：**

```
┌─────────────────────────────────────────────────┐
│                    堆 (Heap)                     │
├───────────────────────┬─────────────────────────┤
│       新生代 (1/3)     │      老年代 (2/3)       │
│  ┌─────────┬─────────┐│                         │
│  │ Eden    │ Survivor ││                         │
│  │   8     │   2     ││                         │
│  │    ┌────┴────┐   ││                         │
│  │    │ S0  │ S1│   ││                         │
│  │    │  1  │  1│   ││                         │
│  │    └────┴────┘   ││                         │
│  └─────────┴─────────┘│                         │
├───────────────────────┴─────────────────────────┤
│                   元空间 (JDK 8+)                │
│                  （方法区实现）                   │
└─────────────────────────────────────────────────┘

比例：Eden:S0:S1 = 8:1:1
新生代:老年代 = 1:2
```

**追问：对象如何进入老年代？**

> 1. **年龄达标**：对象在 Survivor 中每熬过一次 GC，年龄+1，达到 15（默认）进入老年代
> 2. **大对象**：超过某个阈值（-XX:PretenureSizeThreshold）直接进入老年代
> 3. **动态年龄**：Survivor 中同年龄对象大小超过一半，大于该年龄的全部进入老年代
> 4. **空间担保**：Minor GC 前检查老年代最大可用空间是否大于新生代所有对象大小

### 3. 方法区和永久代、元空间的关系？

**答案：**

| JDK 版本 | 方法区实现 | 位置 | 垃圾回收 |
|----------|-----------|------|----------|
| JDK 7 及之前 | 永久代 | JVM 堆 | Full GC |
| JDK 8 及之后 | 元空间 | 本地内存 | Full GC |

```
JDK 7：
┌─────────────────────────────────┐
│            JVM 堆                │
│  ┌─────────┐ ┌────────────────┐ │
│  │ Java 堆  │ │    永久代       │ │
│  └─────────┘ └────────────────┘ │
└─────────────────────────────────┘

JDK 8：
┌─────────────────┐
│     JVM 堆       │  ← Java 堆
│  ┌─────────────┐ │
│  │  Java 堆    │ │
│  └─────────────┘ │
└─────────────────┘

┌─────────────────┐
│    本地内存      │  ← 元空间
│  ┌─────────────┐ │
│  │   元空间    │ │
│  └─────────────┘ │
└─────────────────┘
```

**追问：为什么要用元空间替代永久代？**

> 1. 永久代有固定大小，容易 OOM
> 2. 元空间使用本地内存，大小自动扩展
> 3. 方便与其他语言（如 Groovy、Scala）集成

### 4. 栈帧的结构？

**答案：**

```
每个方法调用创建一个栈帧，栈帧包含：

┌─────────────────────────────────────┐
│              栈帧                    │
├─────────────────────────────────────┤
│  局部变量表                          │
│  ├── 基本类型变量                    │
│  ├── 对象引用                        │
│  └── this 引用（非静态方法）          │
├─────────────────────────────────────┤
│  操作数栈                            │
│  ├── 存放方法执行过程中的中间结果     │
│  └── 计算、方法调用的参数传递         │
├─────────────────────────────────────┤
│  动态链接                            │
│  ├── 指向运行时常量池的方法引用       │
│  └── 支持方法调用的动态绑定           │
├─────────────────────────────────────┤
│  方法返回地址                        │
│  ├── 正常返回：调用者的 PC 寄存器值   │
│  └── 异常返回：异常处理表             │
└─────────────────────────────────────┘
```

## 二、垃圾回收

### 5. 如何判断对象可以被回收？

**答案：**

**方法一：引用计数法**

```java
// 原理：对象被引用 +1，引用失效 -1，为 0 则回收
// 问题：循环引用无法回收

class A { B b; }
class B { A a; }

A a = new A();
B b = new B();
a.b = b;
b.a = a;
a = null;
b = null;
// A 和 B 互相引用，引用计数都不为 0，但实际已无法访问
```

**方法二：可达性分析（JVM 使用）**

```
从 GC Roots 开始搜索，不可达的对象可以回收

GC Roots 包括：
├── 虚拟机栈中的引用（局部变量）
├── 方法区中静态属性引用
├── 方法区中常量引用
├── 本地方法栈中 JNI 引用
├── 同步锁持有的对象
└── JVM 内部引用（基本类型 Class 对象等）
```

**追问：对象被判定为不可达后一定会被回收吗？**

> 不一定。对象至少经历两次标记：
> 1. 第一次标记：不可达，判断是否有必要执行 finalize()
> 2. 如果对象未重写 finalize() 或已执行过，直接回收
> 3. 如果重写了且未执行过，放入 F-Queue，稍后执行
> 4. 第二次标记：finalize() 中对象可以重新与引用链关联，逃脱回收

### 6. 垃圾回收算法有哪些？

**答案：**

**1. 标记-清除（Mark-Sweep）**

```
优点：简单
缺点：
├── 效率不高（两次遍历）
└── 内存碎片

┌─────────────────────────────────┐
│ 标记前：○○●○●●○○●○○●○○●○○       │
│ 标记后：◎◎●◎●●◎◎●◎◎●◎◎●◎◎       │  ◎ = 可回收
│ 清除后：○○  ○  ○○  ○○  ○○       │  出现碎片
└─────────────────────────────────┘
```

**2. 复制算法（Copying）**

```
优点：无碎片、简单高效
缺点：内存利用率低

┌───────────────────────────────────┐
│ 从空间          │    到空间        │
│ ○○●○●●○○●○○●   │                 │
│      ↓ 复制     │                 │
│                 │ ○○○○○○○○       │
│      ↓ 清空     │                 │
│                 │ ○○○○○○○○       │
└───────────────────────────────────┘

新生代优化：Eden:S0:S1 = 8:1:1
只浪费 10% 空间
```

**3. 标记-整理（Mark-Compact）**

```
优点：无碎片、内存利用率高
缺点：效率较低

┌─────────────────────────────────┐
│ 标记前：○○●○●●○○●○○●○○●○○       │
│ 标记后：◎◎●◎●●◎◎●◎◎●◎◎●◎◎       │
│ 整理后：○○○○○○○○               │  紧凑排列
└─────────────────────────────────┘
```

**分代收集：**
- 新生代：复制算法（存活率低）
- 老年代：标记-清除 或 标记-整理

### 7. 常见的垃圾收集器？

**答案：**

```
新生代收集器：
├── Serial：单线程，简单高效，适合客户端
├── ParNew：多线程 Serial，配合 CMS
└── Parallel Scavenge：关注吞吐量

老年代收集器：
├── Serial Old：单线程，标记-整理
├── Parallel Old：多线程，关注吞吐量
└── CMS：并发收集，关注低延迟

G1：分代 + 分区，兼顾吞吐量和延迟
ZGC：并发 + 读屏障，超低延迟（< 10ms）
```

**CMS 收集器流程：**

```
1. 初始标记（STW）：标记 GC Roots 直接关联的对象
2. 并发标记：GC 线程与用户线程并发执行
3. 重新标记（STW）：修正并发标记期间变动的对象
4. 并发清除：并发清除垃圾对象

优点：低延迟
缺点：
├── CPU 敏感（占用线程）
├── 浮动垃圾（并发清理时新产生的垃圾）
└── 空间碎片
```

**追问：G1 和 CMS 的区别？**

| 特性 | CMS | G1 |
|------|-----|-----|
| 内存划分 | 连续新生代 + 老年代 | 多个 Region |
| 回收算法 | 标记-清除 | 复制 + 标记-整理 |
| 回收范围 | 整个新生代或老年代 | 选择价值高的 Region |
| 碎片问题 | 有 | 无 |
| 预测停顿 | 不支持 | 支持 |

---

### 8. Minor GC 和 Full GC 的区别？

**答案：**

| 特性 | Minor GC | Full GC |
|------|----------|---------|
| 回收区域 | 新生代 | 整个堆 + 方法区 |
| 触发条件 | Eden 区满 | 多种条件 |
| 停顿时间 | 短 | 长 |
| 频率 | 频繁 | 较少 |

**Full GC 触发条件：**

```
1. 调用 System.gc()（建议）
2. 老年代空间不足
   ├── 大对象直接进入老年代，空间不足
   ├── 长期存活对象进入老年代，空间不足
   └── 空间担保失败
3. 方法区空间不足
4. CMS 出现 Concurrent Mode Failure
```

## 三、类加载

### 9. 类加载的过程？

**答案：**

```
┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
│  加载   │ → │  验证   │ → │  准备   │ → │  解析   │ → │ 初始化  │
│ Loading │   │ Verify │   │Prepare │   │ Resolve│   │Initialize│
└────────┘   └────────┘   └────────┘   └────────┘   └────────┘
```

**详细过程：**

```
1. 加载（Loading）
   ├── 通过全限定名获取二进制字节流
   ├── 将字节流转化为方法区运行时数据结构
   └── 生成 Class 对象

2. 验证（Verification）
   ├── 文件格式验证（魔数、版本）
   ├── 元数据验证（语义分析）
   ├── 字节码验证（数据流分析）
   └── 符号引用验证

3. 准备（Preparation）
   ├── 为静态变量分配内存
   └── 设置默认初始值（0、null、false）
       注：static final 常量直接赋真实值

4. 解析（Resolution）
   └── 符号引用替换为直接引用

5. 初始化（Initialization）
   └── 执行 <clinit>() 方法，静态变量赋真实值
```

**追问：准备阶段和初始化阶段的区别？**

```java
public class Test {
    static int a = 10;        // 准备：a = 0，初始化：a = 10
    static final int b = 20;  // 准备：b = 20（常量直接赋值）
    static final int c = new Random().nextInt(); 
    // 准备：c = 0，初始化：c = 随机值
}
```

### 10. 双亲委派模型是什么？

**答案：**

```
类加载器层级：

    Bootstrap ClassLoader（启动类加载器）
                ↑
    Extension ClassLoader（扩展类加载器）
                ↑
    Application ClassLoader（应用类加载器）
                ↑
       Custom ClassLoader（自定义类加载器）


双亲委派流程：
1. 收到类加载请求
2. 委派给父类加载器处理
3. 父类无法加载，自己尝试加载
```

**好处：**

```
1. 避免类重复加载
   └── 父类已加载，子类不再加载

2. 保护核心类
   └── 自定义 java.lang.String 无法替换核心类
```

**追问：如何打破双亲委派？**

> 1. **自定义类加载器**：重写 loadClass() 方法
> 2. **SPI 机制**：JDBC 使用线程上下文类加载器
> 3. **OSGi**：模块化加载

```java
public class MyClassLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve) {
        // 1. 自定义加载逻辑
        // 2. 不调用 parent.loadClass()
        // 打破双亲委派
    }
}
```

## 四、调优与排错

### 11. 常用的 JVM 参数？

**答案：**

```bash
# 内存设置
-Xms512m          # 堆初始大小
-Xmx512m          # 堆最大大小（建议与 Xms 相同）
-Xmn256m          # 新生代大小
-Xss256k          # 线程栈大小
-XX:MetaspaceSize=128m    # 元空间初始大小
-XX:MaxMetaspaceSize=256m # 元空间最大大小

# 垃圾收集器
-XX:+UseSerialGC          # Serial + Serial Old
-XX:+UseParallelGC        # Parallel Scavenge + Parallel Old
-XX:+UseConcMarkSweepGC   # ParNew + CMS
-XX:+UseG1GC              # G1

# GC 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log

# OOM 时生成堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=dump.hprof
```

### 12. 内存泄漏和内存溢出的区别？

**答案：**

| 特性 | 内存泄漏 | 内存溢出 |
|------|----------|----------|
| 定义 | 对象无法被回收 | 内存不足以创建对象 |
| 原因 | 代码问题 | 泄漏累积或内存太小 |
| 表现 | 内存逐渐增长 | 直接 OOM |
| 解决 | 找到泄漏点修复 | 增大内存或修复泄漏 |

**常见内存泄漏场景：**

```java
// 1. 静态集合
static List<Object> list = new ArrayList<>();
list.add(new Object());  // 对象永远无法回收

// 2. 未关闭的资源
Connection conn = dataSource.getConnection();
// 忘记关闭

// 3. ThreadLocal 未清理
ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
userThreadLocal.set(user);
// 忘记 remove()

// 4. 监听器未注销
eventBus.register(listener);
// 忘记 unregister
```

### 13. 如何排查 OOM 问题？

**答案：**

```
1. 开启堆转储
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=dump.hprof

2. 使用工具分析
   ├── jvisualvm
   ├── MAT（Memory Analyzer Tool）
   └── JProfiler

3. 分析步骤
   ├── 查看大对象
   ├── 分析对象引用链
   └── 找到泄漏源头

4. 常见命令
   jps      # 查看进程
   jstat    # 查看统计信息
   jmap     # 生成堆转储
   jstack   # 查看线程栈
   jinfo    # 查看参数
```

### 14. JVM 调优的目标和步骤？

**答案：**

**目标：**
- 降低 GC 频率
- 减少 GC 停顿时间
- 避免内存溢出
- 提高系统吞吐量

**步骤：**

```
1. 监控分析
   ├── 使用 jstat、jvisualvm 监控
   ├── 分析 GC 日志
   └── 找出性能瓶颈

2. 确定目标
   ├── 吞吐量优先 → Parallel GC
   ├── 延迟优先 → CMS / G1
   └── 内存占用 → 合理设置堆大小

3. 调整参数
   ├── 堆大小（-Xms、-Xmx）
   ├── 新生代比例（-Xmn、-XX:SurvivorRatio）
   ├── 垃圾收集器选择
   └── GC 策略参数

4. 压测验证
   ├── 压力测试
   ├── 对比调优前后指标
   └── 迭代优化
```
