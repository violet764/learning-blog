# RAG vs SFT 对比分析

在构建基于大语言模型的应用时，我们面临一个关键选择：是使用 RAG（检索增强生成），还是使用 SFT（监督微调），抑或是两者结合？本章将从多个维度深入对比这两种技术，帮助你做出正确的技术选型。

---

## 🎯 核心概念回顾

### RAG 的本质

```
RAG = 检索 + 生成

核心理念：让模型"查资料"再回答

流程：
用户问题 → 检索相关文档 → 文档作为上下文 → LLM 生成答案

特点：
├── 知识存储在外部
├── 实时可更新
├── 可追溯来源
└── 无需训练模型
```

### SFT 的本质

```
SFT = 监督微调

核心理念：让模型"学会"特定知识或技能

流程：
准备数据 → 设计任务格式 → 微调模型 → 模型内化知识

特点：
├── 知识内化到参数
├── 改变模型行为
├── 需要训练资源
└── 效果持久
```

## 📊 多维度对比

### 1. 知识更新

```
场景：公司政策每月更新

RAG 方案：
┌─────────────────────────────────────────┐
│ 知识库 (外部)                            │
│ ├── 政策_v1.md (已过期)                  │
│ ├── 政策_v2.md (已过期)                  │
│ └── 政策_v3.md (最新)      ← 直接更新    │
└─────────────────────────────────────────┘
                    ↓
              RAG 检索
                    ↓
              最新答案

更新成本：分钟级，无需训练


SFT 方案：
┌─────────────────────────────────────────┐
│ 模型参数                                 │
│ ├── 权重包含 v1 知识                      │
│ └── 需要重新微调          ← 需要重新训练  │
└─────────────────────────────────────────┘

更新成本：小时到天级，需要重新训练
```

| 维度 | RAG | SFT |
|------|-----|-----|
| **更新速度** | 实时更新知识库即可 | 需要重新训练 |
| **更新成本** | 低（存储开销） | 高（GPU 计算） |
| **适合场景** | 知识频繁变化 | 知识相对稳定 |

### 2. 知识类型

```
知识类型光谱：

事实性知识 ←─────────────────────────────→ 能力性知识
   │                                           │
   │  "公司报销标准是什么？"                    │  "用专业风格写一封邮件"
   │  "Python 3.11 有什么新特性？"              │  "分析这段代码的时间复杂度"
   │  "最新的税收政策？"                        │  "用医生口吻解释病情"
   │                                           │
   ▼                                           ▼
  RAG 更优                                    SFT 更优
```

| 知识类型 | RAG | SFT | 原因 |
|----------|-----|-----|------|
| **事实性知识** | ✅ 优 | ⚠️ 一般 | RAG 可实时获取，准确度高 |
| **程序性知识** | ⚠️ 一般 | ✅ 优 | SFT 可内化行为模式 |
| **推理能力** | ❌ 弱 | ✅ 优 | RAG 不改变模型推理能力 |
| **风格适应** | ❌ 无效 | ✅ 优 | SFT 可学习特定输出风格 |

### 3. 成本分析

```
成本构成：

RAG 成本：
├── 初始成本
│   ├── 向量数据库部署/云服务费用
│   ├── Embedding 模型（开源免费或 API 费用）
│   └── 文档处理和索引
│
└── 运营成本
    ├── 存储：向量存储费用
    ├── 检索：向量相似度计算
    └── 生成：LLM API 调用

SFT 成本：
├── 初始成本
│   ├── GPU 训练费用（显著）
│   ├── 数据标注/准备成本
│   └── 实验和调优成本
│
└── 运营成本
    ├── 模型托管费用
    └── 推理计算费用
```

### 成本对比表

| 成本项 | RAG | SFT | 说明 |
|--------|-----|-----|------|
| **初始投入** | 💰 低 | 💰💰💰 高 | SFT 需要 GPU 训练 |
| **知识更新成本** | 💰 低 | 💰💰💰 高 | RAG 只需更新文档 |
| **推理成本** | 💰💰 中高 | 💰 低 | RAG 有检索开销 |
| **存储成本** | 💰💰 中 | 💰 低 | RAG 需要向量存储 |
| **人力成本** | 💰 低 | 💰💰 中高 | SFT 需要数据准备 |

### 4. 性能与延迟

```
推理延迟对比：

RAG 流程延迟：
┌──────────┐   ┌──────────┐   ┌──────────┐
│ 查询编码  │ → │ 向量检索  │ → │ LLM生成  │
│  ~10ms   │   │  ~50ms   │   │ ~500ms   │
└──────────┘   └──────────┘   └──────────┘
                    总计: ~560ms

SFT 流程延迟：
┌──────────┐
│ LLM生成  │
│ ~500ms   │
└──────────┘
      总计: ~500ms

差异：RAG 多了检索开销
```

| 性能指标 | RAG | SFT |
|----------|-----|-----|
| **首次响应延迟** | 稍高（检索+生成） | 较低（仅生成） |
| **吞吐量** | 受检索瓶颈限制 | 较高 |
| **冷启动** | 快（无需加载额外模型） | 需加载微调后模型 |

### 5. 数据安全与隐私

```
数据处理位置：

RAG：
┌─────────────────────────────────────────────────────┐
│ 企业内网                                            │
│ ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│ │ 知识库    │  │ 向量存储  │  │ 本地LLM  │          │
│ │ (私有)   │  │ (私有)   │  │ (可选)   │          │
│ └──────────┘  └──────────┘  └──────────┘          │
│                    ↓                                │
│              数据不出内网                            │
└─────────────────────────────────────────────────────┘

SFT：
┌─────────────────────────────────────────────────────┐
│ 训练阶段                                            │
│ 私有数据 → 模型参数（知识已内化）                    │
│ ⚠️ 需要确保训练环境安全                             │
│                                                     │
│ 推理阶段                                            │
│ 模型参数 → 输出（数据已"融入"模型）                  │
│ ✅ 推理时无需原始数据                               │
└─────────────────────────────────────────────────────┘
```

| 安全维度 | RAG | SFT |
|----------|-----|-----|
| **训练阶段数据暴露** | 无需训练 | 需要数据进入训练流程 |
| **推理阶段数据暴露** | 检索文档可能需要发送 | 无需原始数据 |
| **数据删除** | 直接删除文档即可 | 需要重新训练（机器遗忘困难） |

### 组合方案实施

```python
# RAG + SFT 组合架构示例

from langchain.chat_models import ChatOpenAI
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA

class HybridRAGSFTSystem:
    """RAG + SFT 组合系统"""
    
    def __init__(self, finetuned_model_path: str, vectorstore_path: str):
        # 使用微调后的模型
        self.llm = ChatOpenAI(
            model_name=finetuned_model_path,  # 微调后的模型
            temperature=0.1
        )
        
        # 连接向量存储
        self.vectorstore = Chroma(
            persist_directory=vectorstore_path,
            embedding_function=embeddings
        )
        
        # 创建 RAG 链
        self.qa_chain = RetrievalQA.from_chain_type(
            llm=self.llm,
            chain_type="stuff",
            retriever=self.vectorstore.as_retriever(
                search_kwargs={"k": 5}
            ),
            return_source_documents=True
        )
    
    def query(self, question: str) -> dict:
        """查询"""
        result = self.qa_chain({"query": question})
        
        return {
            "answer": result["result"],
            "sources": [
                {
                    "content": doc.page_content[:200],
                    "source": doc.metadata.get("source", "未知")
                }
                for doc in result["source_documents"]
            ]
        }
```

## 💡 最佳实践建议

### 何时优先选择 RAG

```
1. 验证阶段
   └── 先用 RAG 快速验证业务价值

2. 知识型应用
   └── 需要检索外部知识库的场景

3. 数据敏感场景
   └── 数据不能用于训练，但需要查询

4. 快速迭代需求
   └── 业务逻辑频繁变化
```

### 何时优先选择 SFT

```
1. 能力提升需求
   └── 需要模型具备特定能力

2. 风格定制需求
   └── 需要特定的输出风格或格式

3. 高性能要求
   └── 对推理延迟敏感

4. 大规模部署
   └── 长期运营，成本敏感
```

### 组合使用的最佳实践

```
1. SFT 打基础
   ├── 微调领域理解能力
   ├── 学习专业表达风格
   └── 内化常见知识

2. RAG 补充动态信息
   ├── 提供实时数据
   ├── 引用权威来源
   └── 处理长尾问题

3. 持续优化
   ├── 收集用户反馈
   ├── 分析错误案例
   └── 迭代改进两个模块
```
